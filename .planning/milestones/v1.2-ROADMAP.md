## Phase Details

### Phase 13: Production Hardening Fixes
**Goal**: Close four independent production-code gaps documented in `.planning/codebase/CONCERNS.md` — abuse protection on paid AI endpoints, deterministic SSE timer cleanup, app-wide render-error recovery, and removal of unused security-surface packages.
**Depends on**: Nothing (first phase of v1.2; previous milestone shipped 2026-05-08)
**Requirements**: HARD-01, HARD-02, HARD-03, HARD-04
**Success Criteria** (what must be TRUE):
  1. An authenticated user spamming `/api/generate`, `/api/edit-post`, `/api/transcribe`, `/api/carousel/generate`, or `/api/enhance` past the configured per-user limit receives HTTP 429 with a `Retry-After` header, and Gemini is not billed for the rejected requests.
  2. Forcing `sse.sendError` to throw inside a generation request no longer leaks the safety timer — `safetyTimer` is observably cleared on every termination path (success, caught error, thrown error in error handler).
  3. A render error in any descendant of `<App>` shows a user-facing recovery UI ("Something went wrong" + Retry) instead of a blank screen, and logs the error with stack and component info to the browser console.
  4. `package.json` no longer lists `passport`, `passport-local`, `@types/passport`, `@types/passport-local`, `express-session`, `connect-pg-simple`, or `memorystore`; `@octokit/rest` lives under `devDependencies`; `npm install && npm run check && npm run build` all succeed.
**Plans**: 2 plans (13-01, 13-02 — both completed 2026-05-08)

### Phase 14: Wire production crons via HTTP triggers
**Goal**: Make Phase 11 + 12 cron jobs actually fire on Vercel. Add HTTP-triggered endpoints (auth via `CRON_SECRET`) + GitHub Actions schedule. Preserve `node-cron` infrastructure intact so future Hetzner migration is just a config flip.
**Depends on**: Phase 13
**Requirements**: CRON-01, CRON-02, CRON-03, CRON-04
**Success Criteria** (what must be TRUE):
  1. `Authorization: Bearer ${CRON_SECRET}` middleware (`requireCronSecret`) protects three POST endpoints under `/api/internal/cleanup/{trash,purge}` and `/api/internal/billing/run-overage-batch`. Each endpoint invokes the corresponding existing function (`runTrashSweep`, `runPurgeSweep`, `runOverageBillingBatch`) and returns `{ok, trigger:"http", duration_ms, result}`. Missing/wrong header → 401; missing `CRON_SECRET` env → 503.
  2. `.github/workflows/cron.yml` exists with two scheduled jobs: cleanup-sweep (`0 */6 * * *`) hits trash + purge sequentially; overage-batch (`0 0 * * 0`) hits overage batch. Each step uses `set -euo pipefail`, `curl -fsS -X POST --max-time 295`, and reads `${{ secrets.PROD_BASE_URL }}` + `${{ secrets.CRON_SECRET }}`. `workflow_dispatch` enables manual trigger from Actions UI.
  3. The `node-cron` infrastructure (`startCronJobs`, `cron.schedule`) in `server/services/cleanup-cron.service.ts` and the `startCronJobs()` call in `server/index.ts:httpServer.listen` is PRESERVED unchanged — Hetzner migration is a future toggle, not a rewrite.
  4. Architecture documentation (`CLAUDE.md`, `.planning/codebase/ARCHITECTURE.md`, new `docs/production-cron.md`) explains the dual-trigger model: Vercel = HTTP triggers via GitHub Actions; Hetzner = internal `node-cron`. The `cleanup-cron.service.ts` file gains a header doc-comment.
  5. The existing `POST /api/internal/billing/run-overage-batch` handler (currently in `server/routes/billing.routes.ts:649`, unprotected) is moved to the new internal-cron router with `requireCronSecret` applied — same path, same response shape, no contract drift.
**Plans**: TBD

### Phase 15: Cron Verification Harness
**Goal**: Provide an automated, repeatable harness that exercises the three destructive scheduled jobs shipped in Phase 11 and Phase 12 against seeded test data and asserts their observable side effects, replacing the missing human UAT. Verifies the cron functions themselves; the trigger paths (HTTP via Phase 14 / internal via Phase 11+12) are validated implicitly because both invoke the same functions.
**Depends on**: Phase 14
**Requirements**: VRFY-01
**Success Criteria** (what must be TRUE):
  1. Running `scripts/verify-cron-jobs.ts` (or equivalent) seeds three controlled scenarios — past-due posts awaiting trash, posts past `TRASH_RETENTION_DAYS` awaiting purge (with image, thumbnail, slides, and enhancement source files in storage), and `user_billing_profiles` with `pending_overage_micros > 0` — without contaminating real user data.
  2. After invoking `runTrashSweep()`, the harness asserts seeded past-due posts have `trashed_at` set and remain in the database.
  3. After invoking `runPurgeSweep()`, the harness asserts seeded over-retention posts have their DB rows removed AND every associated storage object (post image, thumbnail, slide images, enhancement source) is gone — no orphans left in the bucket.
  4. After invoking `runOverageBillingBatch()`, the harness asserts the expected ledger entries are created for seeded profiles with pending overage and `pending_overage_micros` is reset on success.
  5. The script exits 0 only when all three sweeps produce the expected observable side effects; any deviation produces a non-zero exit and an itemized failure report.
**Plans**: 1 plan (15-01 — planned 2026-05-08, was Phase 14 before reorg)
