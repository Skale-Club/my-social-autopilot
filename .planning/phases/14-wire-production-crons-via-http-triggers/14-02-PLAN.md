This is the body lifted verbatim from CONTEXT.md `<specifics>` → "GitHub Actions workflow shape" and from docs/production-cron.md Path A.
```yaml
name: Production cleanup crons
on:
  schedule:
    - cron: '0 */6 * * *'   # cleanup sweep every 6h
    - cron: '0 0 * * 0'     # overage batch weekly Sunday 00:00 UTC
  workflow_dispatch: {}      # manual trigger for testing

jobs:
  cleanup-sweep:
    if: github.event.schedule == '0 */6 * * *' || github.event_name == 'workflow_dispatch'
    runs-on: ubuntu-latest
    steps:
      - name: Trigger trash sweep
        run: |
          set -euo pipefail
          curl -fsS -X POST "${{ secrets.PROD_BASE_URL }}/api/internal/cleanup/trash" \
            -H "Authorization: Bearer ${{ secrets.CRON_SECRET }}" \
            --max-time 295 \
            -w "\n[HTTP %{http_code}] %{time_total}s\n"
      - name: Trigger purge sweep
        run: |
          set -euo pipefail
          curl -fsS -X POST "${{ secrets.PROD_BASE_URL }}/api/internal/cleanup/purge" \
            -H "Authorization: Bearer ${{ secrets.CRON_SECRET }}" \
            --max-time 295 \
            -w "\n[HTTP %{http_code}] %{time_total}s\n"

  overage-batch:
    if: github.event.schedule == '0 0 * * 0' || github.event_name == 'workflow_dispatch'
    runs-on: ubuntu-latest
    steps:
      - name: Trigger overage batch
        run: |
          set -euo pipefail
          curl -fsS -X POST "${{ secrets.PROD_BASE_URL }}/api/internal/billing/run-overage-batch" \
            -H "Authorization: Bearer ${{ secrets.CRON_SECRET }}" \
            --max-time 295 \
            -w "\n[HTTP %{http_code}] %{time_total}s\n"
```

**Per CONTEXT.md decisions:**
- `workflow_dispatch` enables manual triggering from the GH Actions UI — required so you can smoke-test the wiring after setting up secrets without waiting for the schedule.
- `if:` gates per job ensure that when `workflow_dispatch` fires, BOTH jobs run; when a schedule fires, only the matching job runs. (Otherwise `workflow_dispatch` would fire only one job, defeating the smoke-test.)
- The two cleanup steps in `cleanup-sweep` run sequentially (sequential by default in a single job's `steps:`). Trash sweep MUST run before purge sweep so newly-trashed posts have a chance to age before the purge cutoff is computed against them — though in practice the 30-day retention window makes ordering moot, the order matches the in-process node-cron offset (purge runs 30 min after trash) and the docs/production-cron.md Path B description.
- `--max-time 295` slack is intentional: Vercel maxDuration is 300s (vercel.json:25-30 confirms). Curl needs to return BEFORE Vercel kills the function; 5s gives time for the workflow to read the response body and emit the `-w` log line.
- `-fsS` flags: `-f` fails on HTTP ≥400 (so 401/503 from `requireCronSecret` will fail the step — desirable for visibility); `-s` silences the progress bar; `-S` keeps error messages on failure.

**Out of scope (do NOT add):**
- A timezone field — GH Actions schedule is always UTC.
- A `concurrency:` block — GH Actions schedule is best-effort and overlapping runs are extremely rare; the in-process `overageBatchRunning` lock in cleanup-cron.service.ts handles same-process safety.
- A `permissions: read-all` declaration — this workflow does not touch the repo, only fires curl. Default permissions are fine.
- A pinned action version — this workflow uses no `uses:` actions, only inline `run:` steps, so there's nothing to pin.
- A retry/backoff loop in the curl block — if the endpoint is down we WANT the step to fail loudly; silent retry could mask outages.
